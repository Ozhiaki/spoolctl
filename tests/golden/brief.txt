spoolctl quick brief
Verbs: add, brief, cancel, capabilities, events, list, output, prune, retry, show, status, wait, work.
Jobs move through: canceled, dead, done, failed, queued, running.
JSON mode: every normal verb accepts --json and returns {ok, tool_version, data, meta, warnings, commands, errors}.
SPOOLCTL_DB: queue database path (overridden by --db; default ./.spoolctl/queue.db). --db overrides it.
Typical loop: spoolctl add -- <cmd>; spoolctl work --drain; spoolctl wait <ids>; spoolctl output <id> --stream stdout.
Submit many jobs first, remember their ids, run one worker with work --drain, then wait on all ids. wait exits 6 when any awaited job ends non-success, but the JSON envelope is still ok:true and data.all_succeeded=false.
add supports --key for active queued/running dedup and repeatable --tag KEY=VALUE plus --note for immutable handoff metadata.
list filters by --state and repeatable --tag; show prints full job, attempts, events, key, tags, and note.
events reads the durable job_events ledger: one-shot and --wait return envelopes with meta.pagination.cursor; --follow --json emits raw NDJSON event records only, no control frames.
schema --json exports the envelope, verb data, and raw stream JSON Schemas. capabilities --json describes flags, modes, states, events, env, and exit codes.
Exit codes: 0=success (including empty results), 1=user-input-error, 2=safety-block (refused; a --force form may exist), 3=tool-environment-error, 4=transient-failure (retry after a short delay), 5=conflict (state changed underneath), 6=job-outcome-failure (an awaited job ended non-success).
Use retry for dead/failed jobs, cancel for queued/running withdrawal, prune for old terminal jobs, status for counts/recent dead jobs.
